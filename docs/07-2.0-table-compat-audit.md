# 2.0 Table Compatibility Audit (original research)

*Produced 2026-07-09 on this machine against the live 2.0/Endless Ragnarok install.
This is our own data, not sourced from the community docs — as far as we know, the
first complete per-table map of what the expansion changed.*

## Method

1. Extracted all of `system/table` from the 2.0 `data.i`
   (game updated 2026-07-08) using GBFRDataTools **1.5.1** with the **file list from
   master as of 2026-07-09** (the day Nenkai pushed ER path coverage:
   "filelist: 369354/408165 (90.49%)"). 304 `.tbl` files extracted.
2. Ran `tbl-to-sqlite -v 2.0.2` on **each table in isolation**. The converter validates
   row size against its known schema headers (last updated for ~1.3.x, March 2026), so:
   - **converts OK** → row layout unchanged in 2.0 (1.x schema still fits)
   - **fails with "did not match expected size, it's larger"** → 2.0 **added columns**
     → the table needs a header update upstream, and any mod touching it needs a re-port

## Result: 55 of 304 tables changed

Full lists: [tables-changed-in-2.0.txt](data/tables-changed-in-2.0.txt) ·
[tables-ok-in-2.0.txt](data/tables-ok-in-2.0.txt)

The changed set is essentially the entire high-value gameplay surface:

| Cluster | Changed tables |
|---|---|
| Characters | `chara`, `chara_costume`, `chara_diff`, `chara_invite`, `chara_level_sync` |
| Combat stats | `status`, `skill`, `skill_status`, `limit_bonus`, `limit_bonus_param`, `constant` |
| Enemies | `enemy`, `enemy_status` (+ `_easy`/`_hard`/`_extrem` variants) |
| Items/economy | `item`, `trade`, `gacha_lot`, `reward`, `reward_lot`, `reward_demo`, `dlcpack` |
| Weapons/progression | `weapon`, `ap_open_rank`, `ap_tree_atk`/`_def`/`_wep` |
| Quests/story | `quest_difficulty`, `quest_baseinfo_ex_data`, `story`, `fate_episode`, `chapter_select`, several `story_note_*` |
| UI/misc | `filter*`, `option_*`, `hud_mode`, `menu_unlock`, `tutorial_*`, `dialog`, `badge`, `overlay_map`, `photo_mode_frame`, `collectibles_em`, `bloom`-adjacent |

Notable: `ap_tree_rebuild` did **not** change while the other `ap_tree_*` did; most
`weapon_*` sub-tables (exp, skill levels, status) are unchanged — the new **Weapon
Transcendence** system apparently lives in `weapon.tbl` itself plus the changed
progression tables.

## Caveats

- "Converts OK" ≠ semantically identical — a table can keep its row size while the
  *meaning* of columns changes, and brand-new 2.0 tables that didn't exist in 1.x
  aren't flagged by this method (they'd need a 1.x extraction to diff against; the
  file list itself is also only ~90% complete for ER).
- Failures were not individually inspected beyond the size-mismatch error; a table
  could theoretically fail for a different reason.

## What this enables

- **A working 2.0 research database now**: the 249 unchanged tables converted cleanly →
  `extracted/tables-2.0-partial.sqlite` (gitignored; regenerate with the commands in
  [06-toolchain-setup.md](06-toolchain-setup.md)).
- **A porting triage list**: if a 1.x mod only touches OK-list tables, it can plausibly
  be re-shipped for 2.0 *today* (re-extract → re-apply → convert). If it touches the
  changed list, it's blocked on upstream header updates in
  [GBFRDataTools.Database](https://github.com/Nenkai/GBFRDataTools).
- **An upstream contribution**: the 55-table list is exactly the worklist for updating
  the `.headers` schemas (column research per table). Worth sharing in the Relink
  Modding Discord / as a GBFRDataTools issue.
